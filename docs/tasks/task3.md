# Configuration Module Ambiguities and Decisions

During the implementation of `src/config.rs`, I made a few executive decisions to handle ambiguities or undocumented edge cases in the architecture design:

1. **TOML Array vs Tuple Deserialization:** The architecture specifies `template_points` as `[[0.0, 0.5], ...]`. In `crate::types::GestureTemplate`, `template_points` is defined as a `Vec<(f64, f64)>`. However, `toml` deserialization for arrays of arrays maps more naturally to `Vec<[f64; 2]>`. I used `Vec<[f64; 2]>` for the `GesturePatternConfig` struct in `config.rs` to ensure robust TOML parsing, knowing it will require a quick `.into()` mapping when converting the configuration type to the core `types::GestureTemplate` type.

2. **ActionConfig vs ActionRequest:** The architecture specifies an `ActionRequest` enum in `types.rs` which is fully serializable. The instructions explicitly asked to create an `ActionConfig` enum in `config.rs`. I created `ActionConfig` as a separate type to represent the configuration file schema. In the future, we may want to directly use `ActionRequest` instead of `ActionConfig` to avoid duplication, or implement a `From`/`Into` conversion between them.

3. **Default File Creation Scope:** When `load_config()` is called and `config.toml` doesn't exist, I generate a default `config.toml` and ensure the base `QuickDraw` directory exists. However, I do not automatically generate the `gestures/` subdirectory or a `default.toml` gesture profile at that time. Those will either need to be generated by `load_gesture_profile` if missing, or created when the user saves a new gesture profile via the UI.

4. **Numeric Types in TOML:** The `ARCHITECTURE.md` shows `[gestures.raw]` points as integers (`[[0, 0], [52, 3], ...]`), but the shared `types::GestureCapture` struct defines `points: Vec<(f64, f64)>`. The `toml` crate handles parsing integer literal values into `f64` fields gracefully via `serde`, but it's worth noting this difference between the documented raw data format and the application's internal float representations.

5. **Directory Resolution:** I used `std::env::var("APPDATA")` directly to resolve the configuration directory instead of adding an external dependency like the `dirs` crate. This keeps dependencies minimal and fits the Windows-specific nature of the app (which relies heavily on Win32 APIs).
